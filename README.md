# Описание проекта

Данный pet-проект представляет собой микросервисную систему для создания заказов, назначения курьера, отслеживания его местоположения в режиме реального времени и подтверждения доставки с помощью случайного кода.

Для обеспечения высокой производительности и устойчивости к нагрузке используются асинхронные механизмы взаимодействия между сервисами на базе **Apache Kafka**, **Redis Pub/Sub** и **Spring WebFlux**.

# Архитектура сервисов

## User_service

Отвечает за:

- управление пользователями;
- аутентификацию и авторизацию;
- выдачу JWT access-токенов.

> В текущей версии access-токены имеют увеличенное время жизни исключительно для удобства тестирования.

---

## Payment_service

Отвечает за обработку оплаты заказов.

Данный сервис имитирует внешний платежный сервис, с которым взаимодействие осуществляется по API. Пользователь не имеет прямого доступа к Payment_service — все операции выполняются через Order_service.

На данный момент отсутствует проверка, запрещающая назначение курьера на неоплаченный заказ.

---

## Order_service

Отвечает за:

- CRUD-операции над итемами;
- CRUD-операции над заказами;
- оплату заказов через Payment_service;
- назначение и отзыв курьеров;
- подключение пользователей по WebSocket.

WebSocket используется для передачи пользователю информации о местоположении курьера и данных о заказе в режиме реального времени.

### Последовательность передачи данных

1. Пользователь отправляет запрос на назначение курьера.
2. Courier_service получает событие через Kafka и назначает свободного курьера.
3. После начала доставки курьер каждые 5–10 секунд отправляет POST-запросы со своими координатами в Courier_service.
4. Courier_service публикует данные о местоположении курьера в Kafka.
5. Balancer_order_service считывает сообщение из Kafka.
6. На основании информации, хранящейся в Redis (соответствие пользователя и реплики Order_service), Balancer_order_service отправляет сообщение через Redis Pub/Sub в нужную реплику.
7. Соответствующая реплика Order_service получает сообщение и передает его пользователю по WebSocket.

WebSocket-соединения хранятся в потокобезопасной коллекции `ConcurrentHashMap`.

> В будущем планируется кэширование метода `getCourierOrderUser`. В настоящее время кэшируется только метод `getUsernameForOrder` в Balancer_order_service. Также планируется реализация обработки ошибок доставки.

---

## Courier_service

Отвечает за:

- получение событий назначения заказов из Kafka;
- назначение свободных курьеров;
- изменение статуса выполнения заказа;
- получение текущего выполняемого заказа;
- получение списка всех заказов.

После доставки пользователь сообщает курьеру код подтверждения, после чего курьер изменяет статус заказа с `start` на `done`.

Начальный статус курьера — `AVAILABLE`.

---

## Balancer_order_service

Отвечает за:

- получение событий из Kafka;
- определение реплики Order_service, к которой подключен пользователь;
- передачу сообщений в соответствующую реплику через Redis Pub/Sub.

# Масштабирование

Архитектура предусматривает возможность запуска нескольких реплик каждого сервиса.

# Запуск проекта

Микросервисы запускаются локально.

Через Docker запускаются только инфраструктурные компоненты:

- PostgreSQL;
- Redis;
- Apache Kafka;
- ZooKeeper.

Запуск осуществляется командой:

```bash
docker-compose up -d
```

Команда выполняется из директории Courier_service.

На текущий момент партиционирование Kafka не настроено.

# Kafka

Используемые топики:

- `payment-events`
- `order-events`

# Redis Pub/Sub

Для обмена сообщениями используются каналы, соответствующие наименованию сервисов Order_service.

# Миграции базы данных

Для управления миграциями используется Flyway.

Поскольку Spring Data R2DBC не поддерживает выполнение миграций, дополнительно подключен JDBC и настроена соответствующая конфигурация (Courier_service).

Файлы миграций располагаются в:

```text
resources/db/migration/
```
